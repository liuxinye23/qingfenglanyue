# 安全响应头与浏览器边界审计

## 何时使用

当目标是 Web 站点、管理后台、单页应用、下载中心或含跨域访问的 API 时使用本技能。

## 快速流程

1. 抽样首页、登录页、后台页、静态资源、下载接口、API 接口。
2. 对比各路径的安全头是否一致，尤其是登录态与匿名态差异。
3. 检查 CSP、CORS、缓存头、点击劫持保护、HSTS 与浏览器隔离头。
4. 结合业务上下文判断“缺失”与“配置不当”哪个更关键。

## 重点检查

## 内容安全与脚本执行

- CSP 是否真实生效，还是仅在部分页面存在。
- 是否过度依赖宽松来源、内联脚本豁免或过大的第三方域范围。

## 跨域与跨源

- CORS 是否按资源粒度配置，而不是对敏感接口统一放宽。
- 是否把认证态接口与宽松跨域策略叠加。

## 传输与缓存

- HSTS 是否仅在 HTTPS 主入口存在，子域策略是否明确。
- 登录页、个人中心、下载接口、导出接口的缓存控制是否合适。

## UI 与隔离

- 点击劫持防护是否覆盖后台与支付、配置、审批等敏感页面。
- 浏览器隔离头是否与应用行为匹配，避免只在首页配置。

## 建议工具

## http-framework-test

- 重点看响应头、重定向链、缓存行为和不同页面的头部差异。
- 分别抓匿名态、登录态和高权限态页面。

## nuclei

- 用于补充常见头部缺失与明显错配的模板检测。

## 证据要求

- 保存关键路径的完整响应头摘要。
- 若问题仅出现在某一路由或某一角色，明确写出适用范围。
- 不要只报“缺少某头”，要解释其在该业务中的真实影响。

## 修复建议方向

- 按页面与资源类型制定头部基线，而不是全站复制一份配置。
- 对登录态页面优先收紧缓存、点击劫持与跨域策略。
- 对 CSP、HSTS、CORS 做渐进式验证和回归检查。
